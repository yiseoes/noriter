## 1. 에이전트 총괄

| ID | 에이전트명 | 클래스명 | 스테이지 | 역할 요약 |
|----|-----------|----------|----------|-----------|
| AGT-CTO | CTO | CtoAgent | STAGE 2, 6 | 기술 아키텍처 결정, 디버깅 배정, 수정요청 판단 |
| AGT-PLAN | 기획팀 | PlanningAgent | STAGE 1 | 요구사항 분석, 게임 기획서 작성 |

| AGT-FRONT | 프론트팀 | FrontendAgent | STAGE 4 | 게임 화면·인터랙션 코드 구현 |

| AGT-QA | QA팀 | QaAgent | STAGE 5, 6 | 코드 검증, 버그 리포트, 재테스트 |

## 2. 에이전트별 상세 정의

---

### 2.1 AGT-CTO: CTO 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 기술 총괄. 기획서를 검토하여 기술 아키텍처를 결정하고, QA 실패 시 디버깅 담당 에이전트를 배정한다 |
| **담당 스테이지** | STAGE 2 (Architecture Review), STAGE 6 (Debug 배정) |
| **관련 기능** | NT-AGT-001 (파이프라인), NT-PRJ-006 (수정요청 범위 판단) |

#### 입력 (Input)

| 스테이지 | 입력 항목 | 출처 |
|----------|-----------|------|
| STAGE 2 | 게임 기획서 (plan.json) | AGT-PLAN |
| STAGE 6 | 버그 리포트 (test-report.json) | AGT-QA |
| 수정요청 | 사용자 수정 요구사항 + 기존 산출물 전체 | 사용자 + 시스템 |

#### 산출물 (Output)

| 산출물 | 파일명 | 형식 |
|--------|--------|------|
| 기술 아키텍처 문서 | architecture.json | JSON |

#### architecture.json 구조

```json
{
  "gameType": "single-page",
  "fileStructure": [
    { "path": "index.html", "description": "메인 HTML, Canvas 포함" },
    { "path": "style.css", "description": "게임 스타일시트" },
    { "path": "game.js", "description": "메인 게임 로직" }
  ],
  "technology": {
    "renderer": "Canvas2D | Phaser.js",
    "language": "JavaScript (ES6+)",
    "styling": "CSS3"
  },
  "modules": [
    { "name": "GameEngine", "responsibility": "게임 루프, 렌더링" },
    { "name": "InputHandler", "responsibility": "키보드/마우스 입력" },
    { "name": "ScoreManager", "responsibility": "점수, 레벨 관리" }
  ],
  "notes": "CTO 코멘트 및 기술 결정 사유"
}
```

#### 시스템 프롬프트

```
당신은 게임 개발 스튜디오의 CTO입니다.

## 역할
- 기획팀이 작성한 게임 기획서를 검토하고, 기술 아키텍처를 결정합니다.
- 웹 브라우저에서 실행 가능한 미니게임에 적합한 기술을 선택합니다.

## 제약 조건
- 출력은 HTML5 웹 게임 (HTML + CSS + JavaScript) 만 가능합니다.
- 사용 가능한 렌더링: Canvas 2D API 또는 Phaser.js
- 외부 CDN 라이브러리는 Phaser.js만 허용합니다.
- 파일 수는 최소화합니다 (보통 3~5개).

## 입력
기획서 JSON이 주어집니다.

## 출력
아래 JSON 형식으로 기술 아키텍처 문서를 작성하세요.
{architecture.json 스키마}

반드시 JSON만 출력하세요. 설명이나 마크다운은 포함하지 마세요.
```

#### CTO 특수 역할: 디버깅 배정

QA가 버그를 발견하면 CTO가 버그 유형을 판단하여 수정 담당 에이전트를 배정한다.

```
배정 기준:
├── UI/레이아웃 버그      → AGT-DESIGN + AGT-FRONT
├── 게임 로직 버그        → AGT-BACK
├── 인터랙션/조작 버그     → AGT-FRONT
├── 성능 문제            → AGT-FRONT + AGT-BACK
└── 구조적 문제          → CTO 직접 수정 지시 후 AGT-FRONT + AGT-BACK
```

#### Claude API 설정

| 항목 | 값 |
|------|-----|
| model | claude-sonnet-4-6 |
| maxTokens | 4096 |
| temperature | 0.3 (보수적 판단) |

### 2.4 AGT-FRONT: 프론트엔드팀 에이전트

| 항목 | 내용 |
|------|------|
| **역할** | 게임의 화면 렌더링, 사용자 입력 처리, UI 구현을 담당한다 |
| **담당 스테이지** | STAGE 4 (Implementation, 병렬) |
| **관련 기능** | NT-AGT-001 (파이프라인) |

#### 입력 (Input)

| 입력 항목 | 출처 |
|-----------|------|
| 게임 기획서 (plan.json) | AGT-PLAN |
| 기술 아키텍처 (architecture.json) | AGT-CTO |
| 디자인 스펙 (design.json) | AGT-DESIGN |
| 백엔드 코드 (병렬 완료 시) | AGT-BACK |

#### 산출물 (Output)

| 산출물 | 파일 | 형식 |
|--------|------|------|
| 메인 HTML | index.html | HTML |
| 스타일시트 | style.css | CSS |
| 렌더링·UI 코드 | (game.js 일부 또는 renderer.js) | JavaScript |

#### 시스템 프롬프트

```
당신은 게임 개발 스튜디오의 프론트엔드 개발자입니다.

## 역할
- 게임의 화면(HTML), 스타일(CSS), 렌더링 및 입력 처리(JS)를 구현합니다.
- 디자인 스펙에 따라 시각적으로 완성도 높은 게임을 만듭니다.

## 담당 영역
- index.html: 게임 캔버스, 화면 구조
- style.css: 디자인 스펙의 색상·폰트·레이아웃 적용
- 렌더링: Canvas API 또는 Phaser.js로 게임 오브젝트 그리기
- 입력 처리: 키보드·마우스 이벤트 핸들링
- UI: HUD(점수, HP, 타이머), 화면 전환 (타이틀/게임/게임오버)
- 이펙트: 디자인 스펙의 시각 효과 구현

## 제약 조건
- 외부 이미지 에셋 없이 도형과 CSS만으로 표현합니다.
- 코드는 즉시 실행 가능해야 합니다.
- 백엔드팀이 작성한 게임 로직 코드와 통합됩니다.

## 입력
기획서, 아키텍처, 디자인 스펙 JSON이 주어집니다.
백엔드팀의 게임 로직 코드가 함께 주어질 수 있습니다.

## 출력
각 파일을 아래 JSON 형식으로 출력하세요:
{
  "files": [
    { "path": "index.html", "content": "..." },
    { "path": "style.css", "content": "..." }
  ]
}

반드시 JSON만 출력하세요.
```

#### 시스템 프롬프트

```
당신은 게임 개발 스튜디오의 백엔드(게임 로직) 개발자입니다.

## 역할
- 게임의 핵심 로직을 JavaScript로 구현합니다.
- 프론트엔드팀과 병렬로 작업하며, 통합 가능한 모듈형 코드를 작성합니다.

## 담당 영역
- 게임 루프 (requestAnimationFrame 기반)
- 게임 오브젝트 관리 (생성, 업데이트, 삭제)
- 충돌 감지 (히트박스, 거리 계산)
- 점수·레벨 시스템
- 난이도 조절 로직
- 게임 상태 관리 (시작, 플레이, 일시정지, 게임오버)

## 제약 조건
- 순수 JavaScript (ES6+), 외부 라이브러리 최소화
- 전역 오염 방지 (클래스 또는 모듈 패턴 사용)
- 프론트엔드팀의 렌더링 코드와 통합 가능하도록 인터페이스 명확히 정의

## 입력
기획서와 아키텍처 JSON이 주어집니다.

## 출력
각 파일을 아래 JSON 형식으로 출력하세요:
{
  "files": [
    { "path": "game.js", "content": "..." }
  ]
}

반드시 JSON만 출력하세요.
```

## 3. 에이전트 간 통신 흐름

> 참조: 03_아키텍처 §3.5 MessageBus, NT-AGT-002

```
사용자 요구사항
     │
     ▼
[AGT-PLAN] ──HANDOFF──→ [AGT-CTO]
  plan.json                  │
                             │ HANDOFF
                             ▼
                        [AGT-DESIGN]
                          design.json
                             │
                             │ HANDOFF (병렬 분기)
                    ┌────────┴────────┐
                    ▼                 ▼
              [AGT-FRONT]       [AGT-BACK]
              HTML/CSS/JS        game.js
                    │                 │
                    └────────┬────────┘
                             │ HANDOFF (합류)
                             ▼
                         [AGT-QA]
                      test-report.json
                             │
                    ┌────────┴────────┐
                    │                 │
                  PASS              FAIL
                    │                 │
                    ▼                 ▼
                 [출시]          [AGT-CTO]
                              BUG_REPORT 수신
                              디버깅 배정 결정
                                     │
                                     │ REVIEW_REQUEST
                                     ▼
                              [담당 에이전트]
                              수정 코드 작성
                                     │
                                     │ HANDOFF
                                     ▼
                                 [AGT-QA]
                               재테스트 (최대 3회)
```

### 메시지 타입별 사용 시점

| 메시지 타입 | 발신 | 수신 | 시점 |
|-------------|------|------|------|
| HANDOFF | 모든 에이전트 | 다음 스테이지 에이전트 | 스테이지 완료 시 |
| REVIEW_REQUEST | AGT-CTO | 디버깅 담당 에이전트 | QA 실패 후 배정 시 |
| BUG_REPORT | AGT-QA | AGT-CTO | QA 실패 시 |
| FEEDBACK | 시스템 (사용자 대리) | AGT-CTO | 수정 요청(NT-PRJ-006) 시 |
