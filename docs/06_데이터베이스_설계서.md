# 놀이터 (NoriTer) - 데이터베이스 설계서

> **문서 버전**: v1.0
> **작성일**: 2026-03-24
> **상태**: 초안

---

## 1. 설계 원칙

| 항목 | 원칙 |
|------|------|
| **DBMS** | PostgreSQL 16.x (01_개요서 §5, 03_아키텍처 §7) |
| **ORM** | Spring Data JPA + Hibernate (03_아키텍처 §7) |
| **네이밍** | 스네이크 케이스 (테이블·컬럼), 단수형 테이블명 |
| **PK 전략** | 접두사 + UUID 단축형 (예: `prj_a1b2c3d4`) — 05_API 응답 형식과 일치 |
| **시각 컬럼** | `TIMESTAMP WITHOUT TIME ZONE`, ISO 8601 형식 (05_API §1) |
| **소프트 삭제** | 미사용 — 삭제 시 물리 삭제 + 감사 로그로 이력 보존 (NT-MON-005) |
| **인코딩** | UTF-8 (05_API §1) |

### 3.1 project (프로젝트)

> 참조: NT-PRJ-001~006, API-PRJ-001~007, 03_아키텍처 §4.1

사용자의 게임 생성 요청 1건 (01_개요서 §7 용어 정의).

| 컬럼 | 타입 | 제약 | 기본값 | 설명 |
|------|------|------|--------|------|
| `id` | VARCHAR(20) | PK | - | 프로젝트 ID (접두사: `prj_`) |
| `name` | VARCHAR(100) | NOT NULL | - | 프로젝트명 (NT-PRJ-001: 최대 100자, NT-ERR-V002) |
| `requirement` | TEXT | NOT NULL | - | 사용자 요구사항 원문 (NT-PRJ-001: 최소 10자, NT-ERR-V001) |
| `genre` | VARCHAR(20) | NULL | NULL | 게임 장르 (API-PRJ-001: PUZZLE, ACTION, ARCADE, SHOOTING, STRATEGY, OTHER) |
| `status` | VARCHAR(20) | NOT NULL | 'CREATED' | 프로젝트 상태 (03_아키텍처 §4.1) |
| `current_stage` | VARCHAR(20) | NULL | NULL | 현재 진행 중인 스테이지 (API-PRJ-003 응답) |
| `progress` | INTEGER | NOT NULL | 0 | 진행률 0~100 (NT-MON-002, API-PRJ-003) |
| `debug_attempts` | INTEGER | NOT NULL | 0 | 디버깅 반복 횟수 (NT-AGT-005: 최대 3회) |
| `max_debug_attempts` | INTEGER | NOT NULL | 3 | 최대 디버깅 횟수 (NT-AGT-001: 최대 3회 반복) |
| `created_at` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | 생성 시각 |
| `completed_at` | TIMESTAMP | NULL | NULL | 완료 시각 |
| `updated_at` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | 최종 수정 시각 |

**제약 조건:**

```sql
-- PK
CONSTRAINT pk_project PRIMARY KEY (id)

-- CHECK: 상태값 (03_아키텍처 §4.1 프로젝트 상태 전이도)
CONSTRAINT ck_project_status CHECK (
  status IN ('CREATED', 'IN_PROGRESS', 'COMPLETED', 'REVISION', 'FAILED', 'CANCELLED')
)

-- CHECK: 장르 (API-PRJ-001 genre enum)
CONSTRAINT ck_project_genre CHECK (
  genre IS NULL OR genre IN ('PUZZLE', 'ACTION', 'ARCADE', 'SHOOTING', 'STRATEGY', 'OTHER')
)

-- CHECK: 진행률 범위
CONSTRAINT ck_project_progress CHECK (progress BETWEEN 0 AND 100)

-- CHECK: 디버깅 횟수
CONSTRAINT ck_project_debug CHECK (debug_attempts BETWEEN 0 AND max_debug_attempts)
```

**인덱스:**

```sql
-- 목록 조회: 상태 필터 + 최신순 정렬 (API-PRJ-002: status 필터, sort=createdAt,desc)
CREATE INDEX idx_project_status_created ON project (status, created_at DESC);
```

### 3.2 stage (스테이지)

> 참조: NT-AGT-001 (7단계), 03_아키텍처 §4.2, API-PRJ-003 stages[]

파이프라인 내 하나의 작업 단계 (01_개요서 §7 용어 정의).

| 컬럼 | 타입 | 제약 | 기본값 | 설명 |
|------|------|------|--------|------|
| `id` | VARCHAR(20) | PK | - | 스테이지 ID (접두사: `stg_`) |
| `project_id` | VARCHAR(20) | FK, NOT NULL | - | 소속 프로젝트 |
| `type` | VARCHAR(20) | NOT NULL | - | 스테이지 유형 (03_아키텍처 §4.3) |
| `status` | VARCHAR(20) | NOT NULL | 'PENDING' | 스테이지 상태 (03_아키텍처 §4.2) |
| `agent_role` | VARCHAR(30) | NOT NULL | - | 담당 에이전트 역할 (04_에이전트 §1) |
| `stage_order` | INTEGER | NOT NULL | - | 실행 순서 (1~7) |
| `started_at` | TIMESTAMP | NULL | NULL | 시작 시각 |
| `completed_at` | TIMESTAMP | NULL | NULL | 완료 시각 |
| `artifact_id` | VARCHAR(20) | NULL | NULL | 산출물 ID (완료 후) |
| `retry_count` | INTEGER | NOT NULL | 0 | 재시도 횟수 (NT-AGT-005) |
| `error_message` | TEXT | NULL | NULL | 실패 시 에러 메시지 |

**제약 조건:**

```sql
CONSTRAINT pk_stage PRIMARY KEY (id)

CONSTRAINT fk_stage_project FOREIGN KEY (project_id)
  REFERENCES project(id) ON DELETE CASCADE

-- CHECK: 스테이지 유형 (03_아키텍처 §4.3, NT-AGT-001 7단계)
CONSTRAINT ck_stage_type CHECK (
  type IN ('PLANNING', 'ARCHITECTURE', 'DESIGN', 'IMPLEMENTATION', 'QA', 'DEBUG', 'RELEASE')
)

-- CHECK: 상태 (03_아키텍처 §4.2 스테이지 상태)
CONSTRAINT ck_stage_status CHECK (
  status IN ('PENDING', 'IN_PROGRESS', 'COMPLETED', 'FAILED', 'RETRYING', 'SKIPPED')
)

-- CHECK: 에이전트 역할 (04_에이전트 §1 — STAGE 4 병렬 시 복수 가능)
CONSTRAINT ck_stage_agent CHECK (
  agent_role IN ('CTO', 'PLANNING', 'DESIGN', 'FRONTEND', 'BACKEND', 'FRONTEND,BACKEND', 'QA', 'SYSTEM')
)

-- 유니크: 프로젝트당 스테이지 유형 + 순서 유일
CONSTRAINT uq_stage_project_order UNIQUE (project_id, stage_order)
```

**인덱스:**

```sql
-- 프로젝트별 스테이지 조회 (API-PRJ-003: stages[] 순서대로)
CREATE INDEX idx_stage_project_order ON stage (project_id, stage_order);
```

### 3.3 artifact (산출물)

> 참조: NT-AGT-004, 03_아키텍처 §5.3, 04_에이전트 §2 각 에이전트 산출물, API-PRJ-003 artifacts[]

에이전트가 생산한 산출물의 메타 정보 (01_개요서 §7 용어 정의). 실제 파일은 FileSystem에 저장 (03_아키텍처 §5.3).

| 컬럼 | 타입 | 제약 | 기본값 | 설명 |
|------|------|------|--------|------|
| `id` | VARCHAR(20) | PK | - | 산출물 ID (접두사: `art_`) |
| `project_id` | VARCHAR(20) | FK, NOT NULL | - | 소속 프로젝트 |
| `type` | VARCHAR(20) | NOT NULL | - | 산출물 유형 |
| `agent_role` | VARCHAR(20) | NOT NULL | - | 생성 에이전트 (04_에이전트 §1) |
| `version` | INTEGER | NOT NULL | 1 | 버전 (NT-AGT-004: 수정 시 버전업) |
| `file_path` | VARCHAR(500) | NOT NULL | - | 파일 저장 경로 (03_아키텍처 §5.3 workspace/{projectId}/...) |
| `created_at` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | 생성 시각 |

**제약 조건:**

```sql
CONSTRAINT pk_artifact PRIMARY KEY (id)

CONSTRAINT fk_artifact_project FOREIGN KEY (project_id)
  REFERENCES project(id) ON DELETE CASCADE

-- CHECK: 산출물 유형 (03_아키텍처 §6 ArtifactType enum)
-- PLAN=plan.json, ARCHITECTURE=architecture.json, DESIGN=design.json,
-- CODE=게임 소스, TEST_REPORT=test-report.json
CONSTRAINT ck_artifact_type CHECK (
  type IN ('PLAN', 'ARCHITECTURE', 'DESIGN', 'CODE', 'TEST_REPORT')
)
```

**인덱스:**

```sql
-- 프로젝트별 산출물 조회 (API-PRJ-003: artifacts[])
CREATE INDEX idx_artifact_project ON artifact (project_id, created_at);

-- 프로젝트 + 타입별 최신 버전 조회 (에이전트가 이전 산출물 참조 시)
CREATE INDEX idx_artifact_project_type ON artifact (project_id, type, version DESC);
```

**산출물 유형 ↔ 에이전트 ↔ 파일 매핑 (04_에이전트 §2, 03_아키텍처 §5.3):**

| type | agent_role | file_path 패턴 | 스테이지 |
|------|-----------|----------------|----------|
| PLAN | PLANNING | workspace/{id}/artifacts/plan.json | STAGE 1 |
| ARCHITECTURE | CTO | workspace/{id}/artifacts/architecture.json | STAGE 2 |
| DESIGN | DESIGN | workspace/{id}/artifacts/design.json | STAGE 3 |
| CODE | FRONTEND | workspace/{id}/game/index.html, style.css | STAGE 4 |
| CODE | BACKEND | workspace/{id}/game/game.js | STAGE 4 |
| TEST_REPORT | QA | workspace/{id}/artifacts/test-report.json | STAGE 5 |

### 3.4 log_entry (로그)

> 참조: NT-MON-001, NT-MON-004, API-LOG-001, 02_요구사항 §3 NT-MON-001 로그 종류

실시간 로그 스트리밍(SSE)의 영구 저장본.

| 컬럼 | 타입 | 제약 | 기본값 | 설명 |
|------|------|------|--------|------|
| `id` | VARCHAR(20) | PK | - | 로그 ID (접두사: `log_`) |
| `project_id` | VARCHAR(20) | FK, NOT NULL | - | 소속 프로젝트 |
| `level` | VARCHAR(10) | NOT NULL | - | 로그 레벨 (NT-MON-001 로그 종류) |
| `agent_role` | VARCHAR(20) | NULL | NULL | 발생 에이전트 (시스템 로그는 NULL) |
| `stage` | VARCHAR(20) | NULL | NULL | 발생 스테이지 |
| `message` | TEXT | NOT NULL | - | 로그 메시지 |
| `error_code` | VARCHAR(20) | NULL | NULL | 에러 코드 (05_API §8 에러 코드 체계) |
| `stack_trace` | TEXT | NULL | NULL | 스택 트레이스 (NT-MON-004: 접기/펼치기) |
| `resolved` | BOOLEAN | NULL | NULL | 해결 여부 (NT-MON-004: 미해결/자동복구/수동해결) |
| `resolved_by` | VARCHAR(20) | NULL | NULL | 해결 방법 (AUTO_RETRY 등) |
| `timestamp` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | 발생 시각 |

**제약 조건:**

```sql
CONSTRAINT pk_log_entry PRIMARY KEY (id)

CONSTRAINT fk_log_project FOREIGN KEY (project_id)
  REFERENCES project(id) ON DELETE CASCADE

-- CHECK: 로그 레벨 (NT-MON-001, 03_아키텍처 §6 LogLevel enum)
CONSTRAINT ck_log_level CHECK (
  level IN ('INFO', 'AGENT', 'WARN', 'ERROR', 'AUDIT', 'DEBUG')
)
```

**인덱스:**

```sql
-- 프로젝트별 로그 조회 — 레벨·에이전트 필터 (API-LOG-001 query params)
CREATE INDEX idx_log_project_time ON log_entry (project_id, timestamp DESC);
CREATE INDEX idx_log_project_level ON log_entry (project_id, level);
CREATE INDEX idx_log_project_agent ON log_entry (project_id, agent_role);
```

### 3.6 audit_log (감사 로그)

> 참조: NT-MON-005, API-LOG-003, NFR-03 (최소 90일 보존)

시스템 전체의 감사 로그 (01_개요서 §7 용어 정의). project_id가 NULL일 수 있음 (시스템 설정 변경 등).

| 컬럼 | 타입 | 제약 | 기본값 | 설명 |
|------|------|------|--------|------|
| `id` | VARCHAR(20) | PK | - | 감사 로그 ID (접두사: `audit_`) |
| `event_type` | VARCHAR(30) | NOT NULL | - | 이벤트 유형 (API-LOG-003 eventType) |
| `project_id` | VARCHAR(20) | NULL | NULL | 관련 프로젝트 (시스템 이벤트는 NULL) |
| `description` | VARCHAR(500) | NOT NULL | - | 이벤트 설명 |
| `detail` | JSONB | NULL | NULL | 상세 정보 (API-LOG-003 응답의 detail 객체) |
| `timestamp` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | 발생 시각 |

**제약 조건:**

```sql
CONSTRAINT pk_audit_log PRIMARY KEY (id)

-- FK는 걸지 않음 — 프로젝트 삭제 후에도 감사 로그 보존 (NFR-03: 90일)
-- project_id는 논리적 참조만 유지

-- CHECK: 이벤트 유형 (API-LOG-003, NT-MON-005 기록 대상)
CONSTRAINT ck_audit_event_type CHECK (
  event_type IN (
    'PROJECT_CREATED',     -- 프로젝트 생성
    'PROJECT_DELETED',     -- 프로젝트 삭제
    'PROJECT_STATUS_CHANGED', -- 프로젝트 상태 변경
    'STAGE_STARTED',       -- 스테이지 시작
    'STAGE_COMPLETED',     -- 스테이지 완료
    'STAGE_FAILED',        -- 스테이지 실패
    'ARTIFACT_CREATED',    -- 산출물 생성
    'USER_ACTION',         -- 사용자 조작 (요청, 중단, 재시도 등)
    'SETTING_CHANGED'      -- 설정 변경 (API 키 등)
  )
)
```

**인덱스:**

```sql
-- 감사 로그 목록 조회 (API-LOG-003: 날짜 범위, 이벤트 타입, 프로젝트별 필터)
CREATE INDEX idx_audit_timestamp ON audit_log (timestamp DESC);
CREATE INDEX idx_audit_event_type ON audit_log (event_type, timestamp DESC);
CREATE INDEX idx_audit_project ON audit_log (project_id, timestamp DESC);
```

**보존 정책 (NFR-03):**

```sql
-- 90일 이전 감사 로그 정리 (배치 또는 pg_cron)
-- 주의: project 삭제와 독립적으로 보존해야 하므로 FK 없음
DELETE FROM audit_log WHERE timestamp < NOW() - INTERVAL '90 days';
```

## 5. 테이블 관계 요약

| 관계 | 설명 | 삭제 정책 |
|------|------|-----------|
| project → stage | 1:N (프로젝트당 최대 7개 스테이지) | CASCADE |
| project → artifact | 1:N (스테이지별 산출물) | CASCADE |
| project → log_entry | 1:N (실시간 로그 이력) | CASCADE |
| project → agent_message | 1:N (에이전트 간 통신) | CASCADE |
| project → token_usage | 1:N (토큰 사용량) | CASCADE |
| audit_log → (project) | N:1 논리적 참조 (FK 없음) | 독립 보존 (90일) |

**CASCADE 근거:** 프로젝트 삭제 시 관련 데이터 일괄 삭제 (NT-PRJ-005: 프로젝트 및 산출물, 로그 일괄 삭제).
단, audit_log는 FK를 걸지 않음 — 감사 로그는 프로젝트 삭제와 무관하게 90일 보존 (NFR-03).

## 6. 데이터 생성 흐름 (파이프라인 기준)

> 참조: NT-AGT-001 파이프라인 실행, 04_에이전트 §3 통신 흐름

```
1. 사용자 게임 생성 요청 (API-PRJ-001)
   ├─ INSERT project (status=CREATED)
   ├─ INSERT stage × 6~7개 (status=PENDING)
   └─ INSERT audit_log (PROJECT_CREATED)

2. 파이프라인 시작
   ├─ UPDATE project (status=IN_PROGRESS)
   └─ INSERT audit_log (PROJECT_STATUS_CHANGED)

3. 각 스테이지 실행 (STAGE 1~7 반복)
   ├─ UPDATE stage (status=IN_PROGRESS)
   ├─ INSERT audit_log (STAGE_STARTED)
   ├─ INSERT log_entry × N (실시간 로그)
   ├─ INSERT token_usage (API 호출마다)
   │
   ├─ 성공 시:
   │    ├─ INSERT artifact (산출물 저장)
   │    ├─ INSERT audit_log (ARTIFACT_CREATED)
   │    ├─ INSERT agent_message (HANDOFF — 다음 에이전트에게)
   │    ├─ UPDATE stage (status=COMPLETED)
   │    ├─ INSERT audit_log (STAGE_COMPLETED)
   │    └─ UPDATE project (progress 갱신, current_stage 갱신)
   │
   └─ 실패 시:
        ├─ INSERT log_entry (level=ERROR)
        ├─ UPDATE stage (status=FAILED)
        └─ INSERT audit_log (STAGE_FAILED)

4. QA 실패 → 디버깅 순환 (최대 3회)
   ├─ INSERT agent_message (BUG_REPORT: QA → CTO)
   ├─ INSERT agent_message (REVIEW_REQUEST: CTO → 담당 에이전트)
   ├─ UPDATE project (debug_attempts++)
   └─ 3회 초과 시: UPDATE project (status=FAILED)

5. 파이프라인 완료
   ├─ UPDATE project (status=COMPLETED, progress=100, completed_at)
   └─ INSERT audit_log (PROJECT_STATUS_CHANGED)

6. 수정 요청 (API-PRJ-006)
   ├─ UPDATE project (status=REVISION)
   ├─ INSERT agent_message (FEEDBACK: SYSTEM → CTO)
   ├─ INSERT audit_log (USER_ACTION)
   └─ CTO 판단 후 필요 스테이지만 재실행
```

## 7. 초기 DDL

```sql
-- ============================================
-- 놀이터 (NoriTer) 데이터베이스 초기화 스크립트
-- PostgreSQL 16.x
-- ============================================

-- 1. project
CREATE TABLE project (
    id                  VARCHAR(20)   PRIMARY KEY,
    name                VARCHAR(100)  NOT NULL,
    requirement         TEXT          NOT NULL,
    genre               VARCHAR(20),
    status              VARCHAR(20)   NOT NULL DEFAULT 'CREATED',
    current_stage       VARCHAR(20),
    progress            INTEGER       NOT NULL DEFAULT 0,
    debug_attempts      INTEGER       NOT NULL DEFAULT 0,
    max_debug_attempts  INTEGER       NOT NULL DEFAULT 3,
    created_at          TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    completed_at        TIMESTAMP,
    updated_at          TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT ck_project_status CHECK (
        status IN ('CREATED','IN_PROGRESS','COMPLETED','REVISION','FAILED','CANCELLED')
    ),
    CONSTRAINT ck_project_genre CHECK (
        genre IS NULL OR genre IN ('PUZZLE','ACTION','ARCADE','SHOOTING','STRATEGY','OTHER')
    ),
    CONSTRAINT ck_project_progress CHECK (progress BETWEEN 0 AND 100),
    CONSTRAINT ck_project_debug CHECK (debug_attempts BETWEEN 0 AND max_debug_attempts)
);

CREATE INDEX idx_project_status_created ON project (status, created_at DESC);

-- 2. stage
CREATE TABLE stage (
    id              VARCHAR(20)   PRIMARY KEY,
    project_id      VARCHAR(20)   NOT NULL REFERENCES project(id) ON DELETE CASCADE,
    type            VARCHAR(20)   NOT NULL,
    status          VARCHAR(20)   NOT NULL DEFAULT 'PENDING',
    agent_role      VARCHAR(30)   NOT NULL,
    stage_order     INTEGER       NOT NULL,
    started_at      TIMESTAMP,
    completed_at    TIMESTAMP,
    artifact_id     VARCHAR(20),
    retry_count     INTEGER       NOT NULL DEFAULT 0,
    error_message   TEXT,

    CONSTRAINT ck_stage_type CHECK (
        type IN ('PLANNING','ARCHITECTURE','DESIGN','IMPLEMENTATION','QA','DEBUG','RELEASE')
    ),
    CONSTRAINT ck_stage_status CHECK (
        status IN ('PENDING','IN_PROGRESS','COMPLETED','FAILED','RETRYING','SKIPPED')
    ),
    CONSTRAINT uq_stage_project_order UNIQUE (project_id, stage_order)
);

CREATE INDEX idx_stage_project_order ON stage (project_id, stage_order);

-- 3. artifact
CREATE TABLE artifact (
    id              VARCHAR(20)   PRIMARY KEY,
    project_id      VARCHAR(20)   NOT NULL REFERENCES project(id) ON DELETE CASCADE,
    type            VARCHAR(20)   NOT NULL,
    agent_role      VARCHAR(20)   NOT NULL,
    version         INTEGER       NOT NULL DEFAULT 1,
    file_path       VARCHAR(500)  NOT NULL,
    created_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT ck_artifact_type CHECK (
        type IN ('PLAN','ARCHITECTURE','DESIGN','CODE','TEST_REPORT')
    )
);

CREATE INDEX idx_artifact_project ON artifact (project_id, created_at);
CREATE INDEX idx_artifact_project_type ON artifact (project_id, type, version DESC);

-- 4. log_entry
CREATE TABLE log_entry (
    id              VARCHAR(20)   PRIMARY KEY,
    project_id      VARCHAR(20)   NOT NULL REFERENCES project(id) ON DELETE CASCADE,
    level           VARCHAR(10)   NOT NULL,
    agent_role      VARCHAR(20),
    stage           VARCHAR(20),
    message         TEXT          NOT NULL,
    error_code      VARCHAR(20),
    stack_trace     TEXT,
    resolved        BOOLEAN,
    resolved_by     VARCHAR(20),
    timestamp       TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT ck_log_level CHECK (
        level IN ('INFO','AGENT','WARN','ERROR','AUDIT','DEBUG')
    )
);

CREATE INDEX idx_log_project_time ON log_entry (project_id, timestamp DESC);
CREATE INDEX idx_log_project_level ON log_entry (project_id, level);
CREATE INDEX idx_log_project_agent ON log_entry (project_id, agent_role);

-- 5. agent_message
CREATE TABLE agent_message (
    id              VARCHAR(20)   PRIMARY KEY,
    project_id      VARCHAR(20)   NOT NULL REFERENCES project(id) ON DELETE CASCADE,
    from_agent      VARCHAR(20)   NOT NULL,
    to_agent        VARCHAR(20)   NOT NULL,
    type            VARCHAR(20)   NOT NULL,
    content         TEXT          NOT NULL,
    artifact_ref    VARCHAR(20),
    created_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT ck_message_type CHECK (
        type IN ('HANDOFF','REVIEW_REQUEST','BUG_REPORT','FEEDBACK')
    )
);

CREATE INDEX idx_message_project_time ON agent_message (project_id, created_at);
CREATE INDEX idx_message_project_agent ON agent_message (project_id, from_agent);

-- 6. audit_log (FK 없음 — 프로젝트 삭제 후에도 90일 보존)
CREATE TABLE audit_log (
    id              VARCHAR(20)   PRIMARY KEY,
    event_type      VARCHAR(30)   NOT NULL,
    project_id      VARCHAR(20),
    description     VARCHAR(500)  NOT NULL,
    detail          JSONB,
    timestamp       TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT ck_audit_event_type CHECK (
        event_type IN (
            'PROJECT_CREATED','PROJECT_DELETED','PROJECT_STATUS_CHANGED',
            'STAGE_STARTED','STAGE_COMPLETED','STAGE_FAILED',
            'ARTIFACT_CREATED','USER_ACTION','SETTING_CHANGED'
        )
    )
);

CREATE INDEX idx_audit_timestamp ON audit_log (timestamp DESC);
CREATE INDEX idx_audit_event_type ON audit_log (event_type, timestamp DESC);
CREATE INDEX idx_audit_project ON audit_log (project_id, timestamp DESC);

-- 7. setting
CREATE TABLE setting (
    id              VARCHAR(20)   PRIMARY KEY,
    key             VARCHAR(100)  NOT NULL UNIQUE,
    value           TEXT          NOT NULL,
    updated_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 8. token_usage
CREATE TABLE token_usage (
    id              VARCHAR(20)   PRIMARY KEY,
    project_id      VARCHAR(20)   NOT NULL REFERENCES project(id) ON DELETE CASCADE,
    agent_role      VARCHAR(20)   NOT NULL,
    stage           VARCHAR(20)   NOT NULL,
    input_tokens    INTEGER       NOT NULL DEFAULT 0,
    output_tokens   INTEGER       NOT NULL DEFAULT 0,
    created_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_token_project_agent ON token_usage (project_id, agent_role);
```
