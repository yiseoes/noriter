## 게임 초기 생성 품질 개선 플랜

> **배경**: 게임을 만들 때마다 사후 버그 수정을 반복하는 구조. 매번 떼우기식 대응으로는 품질이 올라가지 않음.
> 테트리스에서 발생한 문제들(canvas 확대, 블록 네모, 레이아웃 엉망)을 계기로 근본 원인을 분석하고, 처음부터 올바른 게임이 나오는 구조를 만들기 위한 플랜.

---

### 근본 원인 분석 — 왜 게임이 처음부터 제대로 안 나오는가

#### 원인 1. 에이전트 간 계약이 선언만 있고, 검증이 없다
CTO가 `architecture.json`에 `shape: Array<Array<Array<number>>>` 라고 써도:
- Backend는 그렇게 구현할 수 있고
- Frontend는 그 계약을 2D 배열로 해석할 수 있음
- **아무도 "두 코드가 실제로 일치하는가"를 확인하지 않음**

테트리스 블록 네모 버그가 정확히 이 케이스: CTO 계약에 rotation shape 타입이 불명확 → Frontend가 잘못 접근 → 모든 블록 네모

#### 원인 2. Canvas 초기화가 시스템 레벨에서 보장되지 않았음
`canvas.width/height`를 설정하는 코드가 CTO 템플릿 + CodeMerger 어디에도 없었음.
→ AI가 당연히 쓰지 않음 → 기본값 300×150 → CSS stretch → 500% 확대
이건 "AI 잘못"이 아니라 **시스템이 보장해야 할 것을 설계에서 빠뜨린 것** (이번 세션에 수정 완료)

#### 원인 3. sharedConstants가 사실상 선택사항
CTO가 `CELL_SIZE`, `BOARD_X` 같은 레이아웃 상수를 sharedConstants에 안 써도 파이프라인이 진행됨.
→ Backend는 24, Frontend는 26을 각자 결정 → 그리드 불일치
sharedConstants가 "정보 전달" 역할인데, 실제로 강제되지 않으니 의미가 반감됨

#### 원인 4. Frontend가 Backend 실제 구현을 못 봄
Frontend는 `architecture.json` 계약서만 보고 작성. Backend가 실제로 어떤 구조로 데이터를 넘기는지 모름.
→ 계약서 해석 차이 → 런타임 버그 → 지금은 `JsRuntimeValidator`가 일부 잡아주지만 완전하지 않음

#### 원인 5. QA가 실제 렌더링을 볼 수 없음
QA agent는 코드 분석만 가능. "이론상 맞는 코드"가 "실제로 눈에 보이는 게임"을 보장하지 않음.
→ 시각적 버그(레이아웃 엉망, 블록 모양 이상)는 QA를 통과함

#### 원인 6. 버그 발견이 너무 늦은 단계에서 일어남
Stage 4 생성 → Stage 5 QA → Stage 6 Debug 순서로 발견.
초기 생성에서 잡을 수 있는 구조적 버그를 3스테이지 지난 후에 발견하는 구조

#### 원인 7. 프롬프트가 "하지 마라" 위주, 올바른 코드 예시가 부족함
금지 규칙은 많지만 "이렇게 해라"는 실제 코드 예시가 적음.
AI는 추상적인 금지보다 **구체적인 올바른 패턴 예시에 훨씬 더 잘 반응**함

```
기획 → CTO → 디자인 → Backend → Frontend (Backend 코드 참조) → 병합 → QA
```

- Frontend agent 컨텍스트에 Backend 생성 코드 포함 → 계약 해석 차이 원천 차단

### 1. 소스코드 탭 내용 미표시 버그 수정

**증상**: 소스코드 탭을 열어도 파일 목록 자체가 비어 있거나 내용이 로드되지 않음

**원인 추적**:
- `GET /projects/{id}/source` 응답 구조: `{ "files": [...] }` 객체
- 프론트 `gameApi.ts`의 `getGameFiles()` 함수가 응답을 배열로 직접 가정 → `files.map()` 호출 시 TypeError
- react-query가 에러를 catch해 `undefined` 반환 → SourceTab에 빈 배열 전달

**수정**:
```typescript
// 수정 전
client.get(...).then(r => r.data.map(f => ...))

// 수정 후
client.get<{ files: {...}[] }>(...).then(r => (r.data.files ?? []).map(f => ({ name: f.path, size: f.size })))
```

### 2. game.js / index.html 편집 버튼 미표시 버그 수정

**증상**: 소스코드 탭에서 `style.css`에서만 편집 버튼이 보이고, `game.js`·`index.html` 탭에서는 편집 버튼이 나타나지 않음

**원인 추적**:
- Spring MVC의 `@GetMapping("/source/{path}")` — path variable이 URL 끝에 오고 `.js`, `.html` 같은 알려진 확장자를 포함하면 Spring이 확장자를 suffix로 인식해 자동으로 제거
- `game.js` 요청 → path variable = `"game"` → `readGameFile(id, "game")` → 파일 없음 → 404
- `style.css`는 `.css`가 알려진 미디어타입이 아니어서 잘리지 않음 → 정상 동작
- 결과적으로 game.js·index.html 파일 내용 로드 실패 → `useGameFiles`가 `"// 파일을 불러올 수 없습니다."` 반환하지만 파일 자체는 존재 → 탭은 보이나 편집 불가 상태

**수정**: GET·PUT 모두 `:.+` 패턴 적용
```java
// 수정 전
@GetMapping("/source/{path}")
@PutMapping("/source/{path}")

// 수정 후
@GetMapping("/source/{path:.+}")
@PutMapping("/source/{path:.+}")
```

### 6. 크로스파일 커플링 감지 기능 구현 (핵심)

- architecture.json 계약서 기반 검사도 검토했으나, 실제 유저가 체감하는 커플링은 파일 간 직접 참조이므로 그 방향으로 전환

```
  → 변경된 CSS에서 #X, .cls 선택자 추출
  → 기존 index.html에 해당 ID/클래스 존재 여부 확인
```

**파싱 로직** (모두 Java Regex):
```java
// HTML ID 추출
Pattern.compile("id\\s*=\\s*[\"']([^\"']+)[\"']")

// HTML 클래스 추출 (space-separated)
Pattern.compile("class\\s*=\\s*[\"']([^\"']+)[\"']")

// JS getElementById 참조
Pattern.compile("getElementById\\s*\\(\\s*[\"']([^\"']+)[\"']\\s*\\)")

// JS querySelector('#id') 참조
Pattern.compile("querySelector\\s*\\(\\s*[\"']#([\\w-]+)[\"']\\s*\\)")

// CSS ID 선택자
Pattern.compile("#([\\w-]+)\\s*[\\{,:\\[]")

// CSS 클래스 선택자
Pattern.compile("\\.([\\w-]+)\\s*[\\{,:\\[]")
```

### 7. 디버깅 과정 기록

#### 경고 배너 미표시 문제 (3단계 디버깅)
**1차 원인 가설**: 백엔드가 warnings를 보내지 않는 것 아닌가?
- 서버 로그 확인: `warnings=5건` 확인 → 백엔드 정상

**2차 원인 가설**: `architecture.json` 파싱 실패
- `architecture.json` 파일을 열어보니 파일 전체가 ` ```json ... ``` ` 마크다운 코드펜스로 감싸져 있었음
- `ObjectMapper.readTree()` → JSON 파싱 실패 → try-catch에서 빈 배열 반환 → warnings 0건
- 해결: 파싱 전 코드펜스 제거 로직 추가

**3차 원인**: React 상태 타이밍 문제
- `onSuccess` 콜백에서 `setSavedFile()` → `setEditMode(false)` → `setWarnings()` 순서로 setState 호출
- `setEditMode(false)`가 re-render를 트리거하는데, 이 시점에 `warnings` state가 아직 이전 값(빈 배열)
- `setWarnings()`가 배칭되지 않고 다음 사이클에 적용되어 경고가 한 프레임 뒤에 세팅됨
- 해결: 별도 state 대신 `useEffect`로 `saveMutation.isSuccess` 변화를 감시해 warnings 세팅

#### 커플링 감지 방향 재정의
초기 구현은 `architecture.json`의 `sharedConstants` (CELL_SIZE, btnY 등)와 실제 코드값을 비교하는 방식이었으나, 실제 유저 의도와 달랐음.
- 유저 의도: "A 파일에서 뭔가 바꾸면 B 파일에도 같이 바꿔야 한다고 알려줘라"
- 실제 커플링: HTML의 element ID를 JS에서 직접 참조하는 구조

### 테스트 구축 중 발생한 기술적 문제들

#### UnnecessaryStubbing 문제 (3회 반복)
Mockito strict mode(`@ExtendWith(MockitoExtension.class)`)는 한 번도 호출되지 않은 stub을 에러로 처리. 해결 원칙:
- 여러 테스트에서 공유하는 setup stub → `lenient().when()` 사용
- 특정 테스트에서만 필요한 stub → `when()` 사용하되, 실제로 호출되는지 확인 필요

#### `readGameFile(any(), eq("game.js"))` vs `readGameFile(any(), any())` 순서 문제
Mockito는 나중에 등록된 stub이 먼저 체크됨(LIFO). `any(),any()` stub을 나중에 등록하면 더 specific한 `eq("game.js")` stub이 죽음. 해결: `any(),any()` stub 제거(default null로 충분)

#### `extractMethodBody()` 메서드 탐색 오류
초기 구현이 `" " + methodName + "("` 패턴으로 탐색 → 메서드 호출부(`artifacts = loadExistingArtifacts(projectId)`)도 매칭. 접근 제한자 prefix(`(?:private|protected|public)\s`) 패턴으로 수정하여 정의부만 탐색.

## 2026-04-07 JWT 인증 + 소유권 격리 구현

### 작업 내용
- Spring Security + JWT(jjwt 0.12.6) 기반 인증 시스템 구축
- User 엔티티, AuthService(BCrypt), AuthController(signup/login/me)
- Project에 userId 필드 추가, 모든 API에 소유권 검증 적용
- 프론트엔드: AuthModal 실제 API 연동, Header 프로필 드롭다운, useAuth 훅

### 트러블슈팅: 소유권 격리의 함정

#### 1. "userId가 없잖아?"

JWT 인증을 구현하고 회원가입/로그인이 잘 동작하는 걸 확인한 뒤, 전체 플로우를 검토했더니 **Project 엔티티에 userId 필드가 아예 없었다.** guestId만 있었기 때문에:

- 로그인 유저가 만든 프로젝트의 주인을 구분할 수 없음
- 프로젝트 목록 조회 시 로그인 유저에게 전체 프로젝트가 보임
- 프로젝트 ID만 알면 아무나 삭제/중단 가능

**해결:** Project에 `userId` 필드 추가, `isOwnedBy(userId, guestId)` 메서드 구현, 모든 변경 API에 소유권 검증 로직(`getProjectWithOwnerCheck`) 적용.

#### 2. "왜 119개야? 120개였는데?"

테스트 전체를 돌렸더니 기존 120개에서 119개로 줄어있었다. `ProjectControllerTest.java`를 전체 재작성할 때 `getProjectDetail_notFound_returns404` 테스트를 누락시킨 것. 테스트 수가 줄었다는 건 커버리지가 빠졌다는 뜻이므로, 항상 전/후 개수를 확인해야 한다.

#### 3. "비동기 파이프라인이 테스트를 망친다"

소유권 통합 테스트에서 프로젝트 생성 후 삭제하는 시나리오가 **간헐적으로 실패**했다. 원인 추적 결과:

- `POST /api/projects`가 프로젝트를 생성하면서 `@Async` 파이프라인을 자동 시작함
- 비동기 스레드가 상태를 `CREATED` → `IN_PROGRESS`로 바꿈
- `IN_PROGRESS` 상태에서는 `canDelete() = false`라 삭제가 409로 거부됨
- 비동기 타이밍에 따라 성공하기도, 실패하기도 하는 flaky test 발생

최초 접근: `waitForDeletableStatus()`로 파이프라인 종료 대기 → JPA 1차 캐시 + 트랜잭션 격리로 비동기 스레드의 커밋을 못 읽는 문제 발생. `@DirtiesContext`로 컨텍스트 격리 시도 → 여전히 간헐 실패.

**근본 해결:** 소유권 테스트의 관심사는 "이 사람 프로젝트인가?"이지, 파이프라인 생명주기가 아님. `createProjectDirectly()` = Repository로 직접 DB에 프로젝트 저장 (파이프라인 없이 CREATED 상태 고정). 테스트 관심사와 비동기 로직을 **완전히 분리**.

결과: 3회 연속 전체 테스트 166개 통과, 재현성 확인 완료.

#### 교훈
- 버그 발생 시 **코드 수정 반복 금지**, 원인부터 추적한 뒤 한 번에 수정
- 테스트는 **하나의 관심사만 검증**해야 한다 (소유권 ≠ 파이프라인)
