## 2026-04-07 JWT 인증 + 소유권 격리 구현

### 작업 내용
- Spring Security + JWT(jjwt 0.12.6) 기반 인증 시스템 구축
- User 엔티티, AuthService(BCrypt), AuthController(signup/login/me)
- Project에 userId 필드 추가, 모든 API에 소유권 검증 적용
- 프론트엔드: AuthModal 실제 API 연동, Header 프로필 드롭다운, useAuth 훅

### 트러블슈팅: 소유권 격리의 함정

#### 1. "userId가 없잖아?"

JWT 인증을 구현하고 회원가입/로그인이 잘 동작하는 걸 확인한 뒤, 전체 플로우를 검토했더니 **Project 엔티티에 userId 필드가 아예 없었다.** guestId만 있었기 때문에:

- 로그인 유저가 만든 프로젝트의 주인을 구분할 수 없음
- 프로젝트 목록 조회 시 로그인 유저에게 전체 프로젝트가 보임
- 프로젝트 ID만 알면 아무나 삭제/중단 가능

**해결:** Project에 `userId` 필드 추가, `isOwnedBy(userId, guestId)` 메서드 구현, 모든 변경 API에 소유권 검증 로직(`getProjectWithOwnerCheck`) 적용.

#### 2. "왜 119개야? 120개였는데?"

테스트 전체를 돌렸더니 기존 120개에서 119개로 줄어있었다. `ProjectControllerTest.java`를 전체 재작성할 때 `getProjectDetail_notFound_returns404` 테스트를 누락시킨 것. 테스트 수가 줄었다는 건 커버리지가 빠졌다는 뜻이므로, 항상 전/후 개수를 확인해야 한다.

#### 3. "비동기 파이프라인이 테스트를 망친다"

소유권 통합 테스트에서 프로젝트 생성 후 삭제하는 시나리오가 **간헐적으로 실패**했다. 원인 추적 결과:

- `POST /api/projects`가 프로젝트를 생성하면서 `@Async` 파이프라인을 자동 시작함
- 비동기 스레드가 상태를 `CREATED` → `IN_PROGRESS`로 바꿈
- `IN_PROGRESS` 상태에서는 `canDelete() = false`라 삭제가 409로 거부됨
- 비동기 타이밍에 따라 성공하기도, 실패하기도 하는 flaky test 발생

최초 접근: `waitForDeletableStatus()`로 파이프라인 종료 대기 → JPA 1차 캐시 + 트랜잭션 격리로 비동기 스레드의 커밋을 못 읽는 문제 발생. `@DirtiesContext`로 컨텍스트 격리 시도 → 여전히 간헐 실패.

**근본 해결:** 소유권 테스트의 관심사는 "이 사람 프로젝트인가?"이지, 파이프라인 생명주기가 아님. `createProjectDirectly()` = Repository로 직접 DB에 프로젝트 저장 (파이프라인 없이 CREATED 상태 고정). 테스트 관심사와 비동기 로직을 **완전히 분리**.

결과: 3회 연속 전체 테스트 166개 통과, 재현성 확인 완료.

#### 교훈
- 버그 발생 시 **코드 수정 반복 금지**, 원인부터 추적한 뒤 한 번에 수정
- 테스트는 **하나의 관심사만 검증**해야 한다 (소유권 ≠ 파이프라인)
- 비동기 로직이 포함된 통합 테스트는 테스트 간 격리에 특히 주의
