# E2E 게임 생성 트러블슈팅 기록 (2026-04-10)

## 프로젝트: "야옹이 도감" (prj_fe083e5d)
- 요구사항: 100마리 고양이 수집 힐링 게임, 8개 맵, 픽셀 아트, 도감
- 총 토큰 사용: 289,380+ (약 $4~5)

## 발견된 문제 + 원인 + 해결 방안

### 1. maxTokens 한도 (최초 실패 원인)
- **증상**: FRONTEND/BACKEND 코드 생성 시 JSON 응답이 중간에 잘림
- **원인**: maxTokens=8192 → 게임 코드가 길면 토큰 한도에 걸려 JSON 미완성
- **해결**: 8192→16384 (Sonnet 최대치). 32768 시도 시 HTTP 400 (모델 한도 초과)
- **프롬프트 개선**: 코드를 간결하게 생성하도록 프롬프트에 "최소한의 코드로 핵심 기능만" 지시 필요

### 2. JSON 래퍼 제거 실패
- **증상**: game.js 파일에 `{ "gameJs": "..." }` JSON 래퍼가 그대로 저장됨
- **원인**: BackendAgent가 JSON 파싱 실패 시 원본 그대로 저장. stripCodeBlock은 마크다운만 벗김
- **해결**: JSON 래퍼에서 gameJs 값 수동 추출 로직 추가
- **프롬프트 개선**: "JSON 없이 순수 JavaScript 코드만 반환" 지시를 백엔드 프롬프트에 강화

### 3. 코드 병합 시 잘림
- **증상**: 백엔드 코드 끝부분이 잘린 채 렌더러 코드와 합쳐짐 (함수 미닫힘, 문자열 미완성)
- **원인**: CodeMerger가 잘린 코드를 검증 없이 병합
- **해결**: 병합 전 코드 유효성 검증 (괄호 매칭, 문자열 닫힘 확인) 추가 필요
- **프롬프트 개선**: 각 에이전트가 독립적으로 실행 가능한 완전한 코드를 생성하도록

### 4. HTML <script> 태그 미닫힘
- **증상**: 분할 생성한 index.html에서 `<script>` 열기만 있고 `</script>` 닫기 없음
- **원인**: 분할 생성 시 HTML을 하나의 호출로 생성하는데 토큰 한도로 잘림
- **해결**: 분할 생성 후 HTML 완성도 검증 (닫는 태그 확인 + 자동 보완)
- **프롬프트 개선**: HTML 생성 시 "반드시 </script></body></html>로 끝낼 것" 명시

### 5. 게임 파일 서빙 엔드포인트 없음
- **증상**: preview에서 game.js/style.css 로드 시 500 에러
- **원인**: preview는 index.html만 서빙, 상대경로로 참조된 js/css 파일은 서빙 안 됨
- **해결**: `/api/projects/{id}/game/{fileName}` 엔드포인트 추가 (MIME type 포함)
- **대안**: 모든 코드를 index.html에 인라인으로 넣는 방식 (단일 파일 게임)

### 6. 분할 전략에서 프롬프트 중복
- **증상**: 분할 생성 시 각 파일마다 기획서+아키텍처+디자인 전체를 다시 보냄
- **원인**: 분할 호출 시 컨텍스트 재전송
- **비용 영향**: 입력 토큰 3배 증가 (23K × 3 = 69K 입력 토큰)
- **해결**: 분할 시 이전 파일은 요약만 전달하거나 캐싱

### 7. 디버그 루프 비용 폭발
- **증상**: QA FAIL → 디버그 3회 → 토큰 15만+ 추가 사용
- **원인**: 디버그 시 전체 코드(74KB)를 프롬프트에 넣어서 입력 토큰 폭증
- **해결 방안**:
  - QA 기준 완화 (CRITICAL만 FAIL, HIGH는 경고만)
  - 디버그 시 변경 필요한 부분만 전달 (전체 코드 대신 diff)
  - 디버그 횟수 기본 1회로 줄이기 (3회 → 1회)
- **프롬프트 개선**: QA에게 "사소한 버그는 무시하고 게임이 실행 가능한지만 판단" 지시

### 8. buildRequestBody JSON 깨짐
- **증상**: HTTP 400 에러 (초기 버전)
- **원인**: String.format으로 JSON 생성 시 한글/특수문자 이스케이프 실패
- **해결**: Jackson ObjectMapper로 교체 (완료)

### 9. 타임아웃
- **증상**: FRONTEND 에이전트 타임아웃 (120초 초과)
- **원인**: 16K 토큰 생성에 2~3분 소요
- **해결**: 120초 → 300초 (완료)

### 10. 단일 파일 전략 권장
- **현재**: index.html + style.css + game.js 분리
- **문제**: preview에서 상대경로 참조 문제, 파일 간 의존성 복잡
- **권장**: **index.html 하나에 CSS/JS 인라인** (단일 파일 게임)
- **프롬프트 개선**: "하나의 HTML 파일에 모든 코드를 포함" 방식으로 변경

## 프롬프트 최적화 방향

### 전체 방향
1. **단일 파일 출력**: HTML+CSS+JS를 하나의 파일로 생성 → 파일 서빙/병합 문제 해결
2. **코드 간결화**: "최소한의 코드로 핵심 기능 구현" 강조
3. **JSON 출력 폐지**: JSON 래핑 없이 순수 코드만 반환하도록 변경
4. **QA 기준 완화**: 실행 가능성 중심 (사소한 버그 무시)
5. **디버그 비용 최소화**: 전체 코드 대신 에러 부분만 전달

### 에이전트별 개선
- **PLANNING**: 요구사항을 토큰 효율적 수준으로 축소 (100마리 → 20마리 등 스케일 조정)
- **CTO**: 단일 파일 아키텍처 강제
- **FRONTEND+BACKEND**: 하나의 에이전트로 통합, 단일 HTML 파일 출력
- **QA**: "게임이 브라우저에서 에러 없이 시작되고 기본 조작이 가능하면 PASS" 기준

### 11. Game 클래스 통째로 누락
- **증상**: game.js에 CAT_DATABASE, ParticleSystem, Renderer는 있지만 Game 클래스가 없음
- **원인**: 백엔드 에이전트가 생성한 코드가 토큰 한도(16384)로 잘려서 Game 클래스 부분이 통째로 유실
- **영향**: 게임 초기화, 루프, 상태 관리, 맵 전환, 클릭 처리 등 핵심 로직 전체 부재 → 게임 실행 불가
- **근본 원인**: 프론트+백엔드를 분리 생성 → 병합하는 구조에서, 각각 토큰 한도에 걸리면 코드 완성도 보장 불가
- **해결**: 단일 파일 전략 필수. 한 번의 호출로 완전한 게임을 생성해야 함

### 12. 렌더러 코드도 잘림
- **증상**: Renderer 클래스 메서드 중간에서 문자열이 잘림 (`ctx.font = 'bold 16px "Noto Sans KR`)
- **원인**: 프론트엔드 분할 생성(renderJs)도 16384 토큰 한도로 잘림
- **영향**: 렌더러 클래스 불완전 → 수동 패치 필요
- **해결**: 단일 파일로 CSS/JS/HTML 통합 생성하면 하나의 완전한 파일로 보장

### 13. 프론트/백엔드 분리 구조의 근본 한계
- **문제**: 현재 아키텍처는 Frontend(렌더링) + Backend(로직)를 분리 생성 → CodeMerger로 병합
- **근본 한계**:
  - 두 에이전트가 서로의 코드를 모르고 독립적으로 생성 → 인터페이스 불일치
  - 각각 토큰 한도에 걸리면 불완전한 코드 병합 → 실행 불가
  - 병합 시 코드 유효성 검증 없음
- **최종 해결**: 단일 에이전트가 하나의 HTML 파일로 전체 게임 생성
  - CSS는 `<style>` 태그 인라인
  - JS는 `<script>` 태그 인라인
  - 파일 서빙/병합/참조 문제 모두 해결

## 최종 결론: 프롬프트 고도화 핵심 원칙

### 원칙 1: 단일 파일 출력
- 모든 게임은 **index.html 한 파일**로 생성
- CSS는 `<style>`, JS는 `<script>` 인라인
- 외부 파일 참조 없음 → 서빙 문제 해결

### 원칙 2: 단일 에이전트 생성
- Frontend/Backend 분리 폐지 → **하나의 에이전트**가 전체 게임 생성
- CodeMerger 불필요 → 병합 오류 해결
- 토큰 16384 한도 안에서 완전한 게임 코드

### 원칙 3: 스코프 제한
- 기획 단계에서 **토큰 효율적 수준으로 스케일 다운**
- 100마리 → 20마리, 8맵 → 3맵 등 자동 축소
- "미니게임" 본질에 맞게 심플하게

### 원칙 4: QA 기준 현실화
- PASS 기준: "브라우저에서 에러 없이 시작 + 기본 조작 가능"
- 사소한 버그(스타일 불일치, 일부 기능 미구현)는 PASS 처리
- 디버그 횟수 3회 → 1회로 축소

### 원칙 5: 비용 안전장치
- 프로젝트당 토큰 상한 설정 (예: 100K 토큰)
- 상한 초과 시 자동 중단 + "비용 한도 도달" 알림
- 디버그 루프 비용 예측 후 사용자 확인 요청

### 최적화 후 예상 비용
- 단일 파일 + 단일 에이전트 + QA 완화: ~50K 토큰 → **~$0.50~1.00** (약 700~1,450원)
