# 프롬프트 문제점 분석 — 고양이도감 케이스

## 문제 1: JSON 래핑 파싱 실패 (심각도: 🔴 Critical)

### 현상
- BackendAgent가 `{"gameJs": "...긴 코드..."}` 형태로 응답을 JSON string으로 래핑
- Jackson이 수만 자의 이스케이프된 문자열(`\\n`, `\\"` 등) 파싱 실패
- 결과: `game.js`에 JSON 래퍼가 그대로 저장됨
- 수동으로 `fix.mjs` 스크립트로 추출해야 했음

### 원인
- 백엔드 프롬프트 지시가 "Return ONLY a valid JSON object: `{\"gameJs\": \"...\"}`"인데, 수만 줄의 JS 코드를 JSON string 안에 집어넣으라는 것 자체가 무리한 요구
- 백엔드 응답이 길어질수록 JSON escape가 깨질 가능성 높아짐

### 해결 방향
- **아키텍처 변경**: `gameJs` JSON 키 방식 제거 → 코드 블록으로 반환 (`\`\`\`javascript ... \`\`\``)
- `JsonParser.stripCodeBlock()` 이미 있으므로 활용
- 또는: "Return ONLY the raw JavaScript code, no JSON wrapper, no markdown"로 변경

## 문제 2: Frontend/Backend 역할 분리 실패 (심각도: 🔴 Critical)

### 현상
- FrontendAgent → Renderer 클래스(렌더링 레이어만)
- BackendAgent → Game 클래스(로직만)
- 실제 생성 결과: BackendAgent가 완전한 게임(Game + Renderer + 초기화코드)을 통으로 생성함
- FrontendAgent 결과가 제대로 활용되지 않음

### 원인
- Claude는 "Renderer만 만들어라, Game은 따로 생성될 것이다"라는 분리 개념을 지키기 어려움
- 실제 게임 개발 관점에서는 Game과 Renderer가 강하게 결합되어 있어서 분리 생성이 어색함

### 해결 방향 (두 가지 선택지)
**Option A: 단일 에이전트 방식** (권장)
- FrontendAgent + BackendAgent를 통합하여 하나의 에이전트가 전체 게임 파일 생성
- 프롬프트: "index.html, style.css, game.js 전체를 생성하라"
- CodeMerger 불필요

**Option B: 분리 유지 시 강제 규칙 강화**
- "YOUR CODE MUST NOT contain class `Game`" 명시
- "YOUR CODE MUST NOT contain `requestAnimationFrame`" 명시
- "This file will be merged with: `game.js` which contains `class Game`" 컨텍스트 제공

## 문제 3: CatDrawer 함수 불완전 생성 (심각도: 🟠 High)

### 현상
- CAT_DATABASE에 24종 고양이 정의 (drawFn 참조 포함)
- CatDrawer 객체에는 7개 함수만 생성됨 (나머지 17개 누락)
- 나머지는 fallback 원형으로만 표시됨

### 원인
- 토큰 제한: 24개 픽셀아트 함수를 모두 생성하면 토큰이 매우 큼
- FrontendAgent의 응답이 잘려서(truncated) 불완전하게 저장됨

### 해결 방향
- 고양이 드로잉 함수를 별도 분할 요청으로 생성 (한 번에 4~6개씩)
- 또는: 드로잉 함수를 CTO/Planning 단계에서 데이터로 미리 정의하고 코드 생성에 포함
- **프롬프트 추가**: "CAT_DATABASE에 정의된 모든 고양이의 drawFn을 반드시 구현해야 한다"

## 문제 4: 초기화 코드 충돌 (심각도: 🟠 High)

### 현상
- CodeMerger가 단순 concat: `BackendLogic + "\n\n" + FrontendRender + "\n\n" + 고정 초기화코드`
- BackendAgent가 자체 초기화코드 포함 (`window.addEventListener('DOMContentLoaded', ...)`)
- 결과: 초기화코드가 두 번 실행, `new Game()` 중복 호출

### 원인
- BackendAgent 프롬프트 "At the end, initialization code: const game = new Game()..."에서 초기화코드 생성 지시
- CodeMerger도 고정으로 `window.addEventListener('load', () => { const game = new Game(); })` 추가
- 두 초기화코드가 충돌

### 해결 방향
- BackendAgent 프롬프트에서 초기화코드 지시 제거
- 단일 에이전트 방식으로 전환하면 자연히 해결

## 문제 5: Game/Renderer 인터페이스 불일치 (심각도: 🟡 Medium)

### 현상
- BackendAgent가 생성한 Game 클래스가 `renderer.render()`, `renderer.emitCollectEffect()` 등을 호출
- FrontendAgent가 생성한 Renderer 클래스에 해당 메서드가 없거나 시그니처 불일치
- 예: `renderer.render(game)` 없음, `CatDrawFunctions` 오참조(→ `CatDrawer`)

### 원인
- 두 에이전트가 서로의 코드를 모르고 별개로 생성
- "Renderer 클래스에 이런 메서드가 있을 것이다"라는 컨텍스트가 부족

### 해결 방향
- 단일 에이전트 방식으로 전환하면 자연히 해결
- 분리 유지 시: CTO 단계에서 인터페이스 명세(메서드 시그니처) 생성 → 양쪽 프롬프트에 포함

## 문제 6: DOM UI vs Canvas UI 충돌 (심각도: 🟡 Medium)

### 현상
- FrontendAgent가 생성한 `index.html`에 DOM 기반 UI 포함 (`#top-bar`, `#bottom-bar`, `#pokedex-panel`)
- BackendAgent가 생성한 game.js는 Canvas로 모든 UI 렌더링
- 결과: DOM 요소들이 canvas 위를 덮어 클릭 이벤트 차단

### 원인
- 두 에이전트가 각자 UI를 구현하려 함
- FrontendAgent: DOM 기반 UI (HTML/CSS)
- BackendAgent: Canvas 기반 UI (JS 렌더링)

### 해결 방향
- **단일 에이전트 방식**: 처음부터 UI 방식을 통일
- Planning 단계에서 UI 방식(DOM vs Canvas)을 명확히 결정
- 프롬프트에 명시: "UI는 전부 Canvas로 렌더링한다, HTML에는 `<canvas>` 태그만 있으면 된다"

## 종합 권고: 아키텍처 변경 (B작업 핵심)

### 현재 파이프라인 문제
```
FrontendAgent (Renderer만) + BackendAgent (Game만) → CodeMerger (concat) = game.js
```
→ 실제로는 둘 다 완전한 게임을 생성하려 해서 충돌

### 권장 파이프라인 (B작업)
```
PlanningAgent → CtoAgent → DesignAgent → GameCodeAgent (통합) → QaAgent
```
- `GameCodeAgent`: 기존 FrontendAgent + BackendAgent 통합
- 한 번의 API 호출로 `index.html` + `style.css` + `game.js` 전체 생성
- 출력 형식: JSON `{ "indexHtml": "...", "styleCss": "...", "gameJs": "..." }`
  - 단, gameJs는 JSON string 내부가 아닌 코드블록으로 분리 요청
  - 또는 3개 파일을 분리된 3번의 호출로 순차 생성

### 프롬프트에 추가해야 할 핵심 규칙
1. "게임 JS는 JSON 래퍼 없이 순수 코드로 반환"
2. "모든 UI는 Canvas로 렌더링 (DOM UI 요소 최소화)"
3. "초기화 코드는 파일 맨 끝 한 번만"
4. "CAT_DATABASE에 정의된 모든 drawFn 함수를 반드시 구현"
5. "응답이 잘릴 경우를 대비해 가장 중요한 핵심 로직을 앞에 배치"
